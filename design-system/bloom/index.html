Communications Aid

The Bloom

A 24-hour radial visualization that maps cooperative rhythm to a clock face. Petals show when and how often — never what. Two layers: what has been settled, and what is still ripening toward a decision a person must land.

surface Daybook kind SVG / JavaScript canvas 480×480 · viewBox centered canonical commit fca80aa
Solid petals are settled — events posted to the ledger. Outlines are open matters, filled 0–100% by ripeness. Colour is the hour of day, not the kind.
#s0

What the bloom communicates

The bloom is a clock. Twelve o'clock is noon; six o'clock is midnight at the bottom; three o'clock is sunset on the right, nine o'clock is sunrise on the left. A petal grows outward from the ring at the hour it happened.

It communicates three things: when the cooperative tends to act (the clock position), how often (petal width), and whether something is still open (outline vs. solid). It never communicates content. What a contribution says, what an exchange transfers, what a reflection names — none of that appears in the bloom.

The bloom does not fill itself. A ripening matter can reach 100% and signal readiness — that is the bloom's job. Writing the authoritative event still takes a named person's landing action. Augmentation, not automation.

Three questions it answers

QuestionHow the bloom answers
When?Clock angle — hour maps directly to position on the 24h ring
How often?Petal density at a given hour; petal width scales with entry weight
How close?Outline fill 0–100% for open matters; solid for settled events

What it does not communicate

ContentWhat a contribution says, what an exchange transfers — absent by design
IdentityWhose petal it is — the bloom is collective, not individual
ValueHow much — amounts, hours, dollars do not appear
OutcomeWhether a decision went well — only that it landed

Geometry

The bloom lives in a square SVG viewport. All positions are computed from four constants: the center point, the ring radius, the horizon offset, and the label radius.

ConstantValueRole
cx, cy240, 240Center of the 480×480 viewport
R148Ring radius — petals root here
H50Horizon vertical offset below center — the day/night line
LBLR + 54 = 202Label radius for cardinal text

The clock transform

Hours map to angles with noon at the top (north), matching a clock face. The transform converts a decimal hour to a radian angle, then to SVG x/y coordinates.

// hour → angle (radians, from north)
function thOf(t) { return (t - 12) * Math.PI / 12; }

// angle → SVG point on ring radius R
function pt(r, t)  { var th = thOf(t);  return { x: cx + r*Math.sin(th), y: cy - r*Math.cos(th) }; }
function ptA(r, a) {                     return { x: cx + r*Math.sin(a),  y: cy - r*Math.cos(a) }; }

Horizon and arc

The horizon is a dashed horizontal line at y = cy + H. It marks where the circle intersects this line — sunrise on the left, sunset on the right. The sunrise and sunset hours are derived from geometry, not from a calendar API.

// derived from circle geometry
var dxh     = Math.sqrt(R*R - H*H);          // half-width of horizon chord
var thR     = Math.atan2(dxh/R, -H/R);       // angle to horizon intersection
var sunset  = 12 + thR / (Math.PI / 12);     // ≈ 20.1h in summer geometry
var sunrise = 12 - thR / (Math.PI / 12);     // ≈ 6.4h

// horizon line
x1: cx - dxh,  y1: cy + H
x2: cx + dxh,  y2: cy + H

Six arc segments span sunrise to sunset along the ring, each a different sunset color stepping rose→coral→amber→amber→coral→rose, visualizing the arc of daylight.

Ramp definition

The ramp is a list of [hour, RGB] control points. Any hour between two points is linearly interpolated in RGB space. The array is symmetric around noon.

var GOLD   = [234,182,104];
var AMBER  = [224,154, 82];
var CORAL  = [209,117, 90];
var ROSE   = [180, 92,124];
var VIOLET = [110, 84,148];

var ramp = [
  [ 0,       VIOLET ],
  [ sunrise, ROSE   ],   // ≈ 6.4 — derived from horizon geometry
  [ 7,       CORAL  ],
  [ 10.5,    AMBER  ],
  [ 12.5,    GOLD   ],
  [ 15,      AMBER  ],
  [ 18,      CORAL  ],
  [ sunset,  ROSE   ],   // ≈ 20.1 — derived from horizon geometry
  [ 21.5,    VIOLET ],
  [ 24,      VIOLET ]
];
function colorAt(t) {
  t = ((t % 24) + 24) % 24;
  var a = ramp[0], b = ramp[ramp.length-1];
  for (var i = 0; i < ramp.length-1; i++) {
    if (t >= ramp[i][0] && t <= ramp[i+1][0]) { a = ramp[i]; b = ramp[i+1]; break; }
  }
  var f = (b[0] === a[0]) ? 0 : (t - a[0]) / (b[0] - a[0]);
  var c = [0,1,2].map(function(k) {
    return Math.round(a[1][k] + (b[1][k] - a[1][k]) * f);
  });
  return 'rgb(' + c[0] + ',' + c[1] + ',' + c[2] + ')';
}

Petal algorithm

Each petal is a teardrop shape built from two cubic Bézier curves that share a root on the ring and meet at a tip. The shape is generated procedurally from three parameters: angle, length, and width.

ParameterMeaningSource
aAngle (radians) — clock positionthOf(entry.h) + kind.offset
LLength — how far the petal extends15 + entry.weight × 8
WWidth — half-width at widest point4.5 + entry.weight × 2.2

Bezier teardrop construction

Two unit vectors are computed from the angle: an axial vector (pointing along the petal) and a perpendicular vector (pointing across it). Control points are placed at fixed fractions of L along the axis, offset by ±W along the perpendicular.

function petalPath(a, L, W) {
  // unit vectors: axial (along petal) and perpendicular (across)
  var ax = Math.sin(a),  ay = -Math.cos(a);  // axial
  var px = Math.cos(a),  py =  Math.sin(a);  // perpendicular

  var B = ptA(R, a);  // root point on ring

  // helper: point offset from root by (al) along axis, (pe) perpendicular
  function P(al, pe) {
    return { x: B.x + ax*al + px*pe,
             y: B.y + ay*al + py*pe };
  }

  var T   = P(L,      0      );  // tip
  var cA1 = P(L*0.30,  W     );  // right side, inner control
  var cA2 = P(L*0.86,  W*0.42);  // right side, outer control
  var cB1 = P(L*0.86, -W*0.42);  // left side, outer control
  var cB2 = P(L*0.30, -W     );  // left side, inner control

  // two cubic bezier curves: root → tip (right), tip → root (left)
  return 'M '  + B.x  + ' ' + B.y  +
    ' C ' + cA1.x + ' ' + cA1.y + ' ' + cA2.x + ' ' + cA2.y +
        ' ' + T.x   + ' ' + T.y   +
    ' C ' + cB1.x + ' ' + cB1.y + ' ' + cB2.x + ' ' + cB2.y +
        ' ' + B.x   + ' ' + B.y   + ' Z';
}

Two layers

The bloom renders two overlapping layers. Layer one (settled) is drawn first; layer two (ripening) sits on top. Both use the same petal shape and color math — the visual distinction is fill vs. outline.

Layer 1

Settled bloom

Petals filled with the hour's color at opacity: var(--petal) — 0.5 in dark mode, 0.64 in light. These are events that have been posted to the ledger: confirmed, witnessed, authoritative.

Layer 2

Ripening matters

Outline petals with a semi-transparent fill at ripe × 0.55 opacity. The outline is drawn with a stroke that thickens as ripeness increases: stroke-width: 1 + ripe × 2. A fully ripe matter is unmistakable.

Settled rendering

// CSS
.petal .lobe { opacity: var(--petal); }  /* 0.5 dark / 0.64 light */

// JS — solid filled path
var p = createPath(petalPath(a, L, W));
p.style.fill = colorAt(entry.h);
p.classList.add('lobe');

Ripening rendering

// each matter has a ripe score 0..1
// fill: the hour's color at reduced opacity shows how close
var fp = createPath(petalPath(a, L, W));
fp.style.fill    = colorAt(matter.h);
fp.style.opacity = (matter.ripe * 0.55).toFixed(3);

// outline: stroke only, width grows with ripeness
var ed = createPath(petalPath(a, L, W));
ed.style.fill         = 'none';
ed.style.stroke       = colorAt(matter.h);
ed.style.strokeWidth  = (1 + matter.ripe * 2).toFixed(2);
ed.style.opacity      = '0.6';

Private entries (outline only)

Private settled entries render as dashed outlines with no fill — the petal shape is visible but deliberately not solid. They count as part of the rhythm without fully revealing the moment.

// private settled entry: dashed stroke, no fill
.petal.private .lobe {
  fill: none;
  stroke: currentColor;
  stroke-dasharray: 2 3;
  opacity: 0.4;
}

Entry kinds and angular offsets

Four kinds of entry populate the bloom. All four share the same clock face and color ramp. They are distinguished by a small angular offset from the hour angle — separating petals that would otherwise stack exactly on top of each other when they occur at the same time.

reflection
offset: −0.04 rad
window: first light & evening
Narrative — not posted to the ledger. Public by author choice, no circle assent needed. Always private until landed.
contribution
offset: −0.014 rad
window: working hours
Labor logged for the program budget. Self-confirmed by the member. Requires member's own landing action.
exchange
offset: +0.014 rad
window: mid-morning
Transfers and allocations of value between vaults. Requires the circle's assent before landing.
clearing
offset: +0.04 rad
window: night
Settlements between vaults at day's end. Both peers must confirm before the petal goes solid.

Offset in context

The angle used to place a petal is thOf(hour) + kind.offset. At 0.04 radians, the outermost kinds (reflection and clearing) sit about 5 minutes of clock arc apart from their neighbors. Enough to be visually distinct without obscuring the hour reading.

Data shape

The bloom expects two arrays: one of settled entries grouped by kind, and one of open matters. Both are rendered on each draw call.

Settled entries

// one object per kind, with a data array of settled entries
var types = [
  {
    key:    'contribution',
    label:  'Contribution',
    off:    -0.014,
    window: 'working hours',
    desc:   'labor logged for the program budget',
    data: [
      { h: 9,    w: 1 },   // h = hour (decimal), w = weight (integer)
      { h: 14,   w: 2 },
      { h: 15.5, w: 3 }
    ]
  },
  // … reflection, exchange, clearing
];

Open matters

var matters = [
  {
    id:     'm1',
    kind:   'exchange',          // maps to a kind for angular offset and color
    h:      11.5,                // hour (decimal)
    ripe:   0.78,                // 0..1 ripeness
    awaits: 'the circle\'s assent',
    face:   'allocation $1,500 → LearnVibe.Build',
    cite:   'Bylaws §6',         // bylaw section authorizing this event type
    act:    'signal'             // 'confirm' | 'post' | 'signal' | null
  }
];

Landing — from ripening to settled

When a person takes the landing action, the matter is removed from the ripening layer and a new settled entry is appended to its kind's data array at the same hour. The petal goes solid immediately.

function land(matter) {
  var kindObj = byKey(matter.kind);
  // find an existing petal nearby (within 24 minutes)
  var near = kindObj.data.filter(function(d) {
    return Math.abs(d.h - matter.h) < 0.4;
  })[0];
  if (near) near.w += 1;               // widen existing petal
  else kindObj.data.push({ h: matter.h, w: 2 });  // new petal
  matters = matters.filter(function(x) { return x.id !== matter.id; });
}

Light mode petals are slightly more opaque because the light background provides less contrast. The value was tuned so petals read clearly in both modes without appearing heavy.

Design principles

Colour is the hour. Not the kind. This is the load-bearing decision. If color were per-kind, the bloom would become a pie chart — reading kind distribution, not rhythm. By binding color to time, it stays a clock.

The four principles

PrincipleExpression in the bloom
Time over kindColor encodes hour, not entry type. Kind is encoded only in angular offset (<5 min of arc).
Rhythm over contentThe bloom shows when and how often. Never what. Content is in the entry; the bloom is its silhouette.
Signal, don't actA ripening matter can reach 100% and become unmistakably urgent. It still waits for a person to land it.
Collective, not individualThe bloom aggregates the whole cooperative. No member is singled out. Whose petal — absent by design.

On the absence of axes

The bloom has no y-axis, no legend for scale, no numerical labels. Petal size encodes relative weight — heavier entries have wider petals — but there is no key translating size to hours or dollars. This is deliberate. Precision belongs in the ledger. The bloom conveys pattern.

On private entries

A private settled entry renders as a dashed outline. It says: something happened at this hour. It withholds what. The cooperative's rhythm is visible; its contents remain held. This is the bloom's way of honoring the difference between the cooperative's shared temporal life and each member's private authorship.
