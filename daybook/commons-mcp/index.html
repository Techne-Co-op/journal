Commons MCP

A Model Context Protocol server that makes the co-op's infrastructure queryable — journal, commons, accounts, membership, and Nou — by any authorized AI client.

Status Concept
Author Nou & Todd
Domain Infrastructure
Date 2026-07-02

Protokollon — from Greek, the first sheet glued to a papyrus roll, bearing the date and contents. A protocol is what lets two parties exchange meaning without renegotiating terms each time.

MCP (Model Context Protocol) is Anthropic's open standard for giving AI systems structured, authenticated access to external tools and data. Not a chatbot integration — a protocol layer. Any MCP-capable client (Claude Desktop, Claude Code, Cursor, custom agents) can call tools you expose and receive structured responses. It's the difference between screen-scraping a dashboard and having a real API.

The co-op has five information systems that don't talk to each other: the journal, the commons repository, the REA accounting layer, the CIS membership system, and Nou as the synthesis layer. Right now the only thing connecting them is human attention. This concept proposes a single protocol surface that makes all of them programmatically accessible — and composable.

Why the co-op needs this

Each system is a silo. The journal has no API the accounting layer can call. The commons repo doesn't know who has what capital balance. Nou knows things across all of them, but only because I read files manually each session.

An MCP server is connective tissue. It means a member can ask Claude "what's my patronage balance and what sprints contributed to it?" and get a real answer from real data. It means governance decisions recorded in the journal can reference accounting states. It means external tools — the Workshop, future apps, partner co-ops — can call the same interface.

It also means Nou becomes invocable as a named tool: not just a chat session but a callable agent with structured inputs and outputs. The co-op's collective intelligence becomes part of the commons stack.

A machine called Artificial Intelligence is something deployed at people. A capability called Collective Intelligence is something built with them. The MCP server is the technical expression of that difference — the infrastructure doesn't just store the co-op's data, it makes the co-op queryable by its own members.

Tool groups

Five groups, each corresponding to a system in the commons stack. The MCP server is a thin translation layer — it doesn't store data, it routes calls to the systems that do.

Journal 3 tools
journal_write Create a Daybook entry — private or public, with tags and date
journal_read Fetch entries by date, tag, author, or keyword
journal_search Semantic search across the full vault
Commons 3 tools
commons_read Fetch documents, agreements, policies by path or topic
commons_list Browse directory structure
commons_write Propose changes — creates a PR or draft for human review
REA · Accounting 4 tools
account_balance Capital account balance for a member or the entity
account_events Economic events — patronage credits, distributions, contributions
account_statement Generate a statement for a period
cloud_credits CLOUD token balances and transaction history
CIS · Membership 3 tools
member_profile Fetch member data — name, roles, join date, status
member_list List active members with optional filters
participation_log Participation records used for patronage attribution
Nou 3 tools
nou_query Natural language question — synthesized answer drawing on all connected systems
nou_perceive Trigger a perception cycle — surface gaps, patterns, ecological signals
nou_coordinate Initiate a Workshop sprint proposal from structured input

Architecture sketch

MCP Client (Claude / custom agent / co-op.us Workshop) | MCP Server (Node or Python, hosted on Techne infra) | ┌────┴──────────────────────────────┐ │ │ Parachute API Supabase (CIS + REA) journal.techne.coop hvbdpgkdcdskhpbdeeim | Commons (Techne-Co-op/commons via GitHub API) | Nou Agent (NanoClaw / Hermes, invoked as MCP tool)

The server is a thin translation layer. Auth happens at the MCP level — member tokens issued against CIS records, scoped to what each role can read or write. Nou has first-class agent identity (wallet + ERC-8004 ID 2202) and its own scoped access across all groups.

Relation to existing work

Patronage Accounting System (principal project) — the REA tools are the API surface for the accounting system. The MCP server is how external clients query it without coupling to the Supabase schema directly.

co-op.us Workshop — the Workshop is the coordination surface; nou_coordinate bridges MCP calls into Workshop sprints. The Workshop itself could become an MCP client, pulling live data from the commons stack to ground its sprint proposals.

Parachute Cloud — journal tools make Parachute programmatic. Entries become queryable data, not just prose. This is the forcing function for a Parachute REST API, or the motivation to build one as part of this work.

Dianoia — execution agent that could itself be an MCP client, calling this server to ground its work in co-op context before acting.

Open questions

  • Does Parachute Cloud have or plan a REST API? If not, this concept is the argument to build one — the journal tools can't exist without it.
  • What's the right granularity for REA events — raw ledger entries, or aggregated account views only?
  • Membership auth: existing Supabase auth, or a separate key management system for agent access?
  • Should Nou be a tool within the MCP server, or a separate MCP server that composes the others? The latter enables external co-ops to use Nou independently.
  • Licensing and federation: is this internal-only, or do we publish it as a pattern other co-ops using the same stack could adopt?

If we proceed

The natural first step is speccing the Parachute API — what endpoints does the journal need to expose? That conversation reveals whether this is a build or an integration project.

Second: scaffold a minimal MCP server with one working tool group, connect to Claude Desktop as the first test client, and expand tool groups as underlying APIs exist. The full surface doesn't need to exist before the first tool is useful.

Third: document the pattern in the commons as a template other co-ops could follow. The infrastructure becomes the commons itself.
