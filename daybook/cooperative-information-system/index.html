Cooperative Information System

A database infrastructure for a member-owned organization — tracking who belongs, what they contributed, and what the cooperative owes them.

Topic Data Infrastructure
Audience General
Date 2026-07-02

Every organization needs to know who its members are, what they've done, and what they're owed. Most organizations manage this in spreadsheets, email archives, and accounting software that don't talk to each other.

The Cooperative Information System (CIS) is the infrastructure that answers these questions for RegenHub, LCA — with one additional requirement that most member databases don't have: it must not just record facts, but make them legible to the members themselves, and make the rules that govern those facts traceable back to the governing documents that authorized them.

"It is not a marketing site, a social platform, or a CRM. It is the book of record and the interface through which the cooperative operates itself."

This is a more demanding requirement than it sounds. It means the database doesn't just store data — it enforces the bylaws, and each enforcement rule must cite the bylaw section that authorizes it. The code and the law must remain coherent with each other.

The REA foundation

The CIS is built on REA ontology — a framework from accounting theory that organizes economic information around four primitives: Agents, Resources, Events, and Agreements.

AgentsAny participant: members (natural persons), the cooperative itself (a legal entity), and AI agents like Nou and Dianoia — all first-class in the same table, distinguished by kind
AgreementsBinding commitments — bylaws, membership agreements, grant letters, board motions — with stable addressable sections and a versioning chain so amendments never erase prior text
ResourcesEconomic things with value — capital account balances, equity shares, hub seats, cloud credits — each owned by an agent and governed by an agreement
EventsThings that happen — contributions approved, roles assigned, capital allocated, votes cast — recorded in an append-only log that is both the audit trail and the operational database

The power of this foundation is that every economic and governance action speaks the same language. A role assignment and a capital contribution are both events in the same table, with the same structure: who did it, what changed, when, and under what authority. There is one audit trail, not four.

Four membership classes

RegenHub, LCA has four classes of member, each with different rights and economic relationships:

Cooperative Member
Class One · Labor Member

Full voting (one vote each), patronage eligible, $100 one-time share + $100/year dues. The governing class.

Co-working Member
Class Two · Hub Member

Access to the physical hub and programs. No voting, no patronage. $250–$500/month depending on level.

Community Participant
Class Three

Program attendees, community members. No voting, no patronage. $25–$100/month tiers.

Investor Member
Class Four · Capital Provider

Capital contribution only. No voting unless co-held with Class One. No patronage, no workspace access.

The CIS tracks membership status as a time-bounded role assignment — not a flag on a profile. A member has the patron_member role from their admission date to their departure date, and every transition is an immutable event. The question "was Alice a voting member on March 15th?" is answerable from first principles.

Row-level security: law as code

The CIS enforces access control at the database level using row-level security (RLS) — a PostgreSQL feature that lets you define which rows a user can read or write based on their role. Every table has these policies. There are no exceptions and no bypass paths for application code.

Each policy carries a comment that names the bylaw section or governance document that justifies it. This is called the "cite-as-you-enforce" pattern. An example:

-- members_read_directory
Patron and Investor members may read other active members' display names and emails. Authority: bylaws §1.7, §2.9 (member information rights, role scopes).

This means the database schema is a living reference to the cooperative's bylaws. When bylaws change, the policies must change to match, and the citations must be updated. The code and the law stay coherent because the code quotes the law.

The default posture is deny: no policy means no access. There are fifteen roles in the system (patron member, investor member, suspended member, director, officer, committee member, administrator, applicant, guest, Nou, Dianoia, hub applicant, hub visitor, hub member, liaison), and each sees exactly what their role permits, no more.

Contribution lifecycle and patronage attribution

Every member contribution moves through a five-stage lifecycle before it affects patronage calculations:

→Created — member logs raw input (hours worked, revenue generated, capital contributed, community activity)
→Submitted — member sends for review with supporting context
→Validated — system checks format and required metadata
→Valued — a credit amount is assigned in dollars or CLOUD credits
→Approved — coordinator review locks the contribution for allocation calculation

Each transition is an immutable event in the audit log. Once a contribution is approved, it cannot be silently altered — only formally amended through a documented process. This immutability is the foundation of the allocation audit trail: any member can trace their patronage allocation back through the specific approved contributions that produced it.

The CIS also tracks member standing as a function of contribution history: five or more contributions reaches Contributor status, twenty reaches Steward, fifty reaches Principal. These tiers affect what tools and roles are available, not the patronage formula itself — patronage is purely proportional, not threshold-based.

Bylaws as queryable data

One of the CIS's less obvious features is that the bylaws themselves live in the database. The agreement_sections table holds every clause of the bylaws as a row, with a stable address — bylaws:v2:art-1.sec-4 — that survives amendments. When a section is amended, a new row is created and linked to the old one via a superseded_by field. Prior text is never deleted.

This means a member can ask "what governs my right to see other members' email addresses?" and get the answer programmatically — the database can return the exact bylaw section, its text, and its effective date. In a later phase, these sections will have vector embeddings, allowing semantic search: "what are my rights if I want to inspect the cooperative's financial records?"

The deeper aspiration is that the cooperative's governing documents should be as accessible as its operational data. A filing cabinet of PDFs governs nominally; a queryable knowledge base governs in practice.
